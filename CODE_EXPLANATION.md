# Code Explanation: V5 Hybrid Vision Pipeline

This document explains the core logic found in `app.py`, specifically how the 3-Stage Hybrid Computer Vision Architecture operates on a frame-by-frame basis.

The pipeline was designed to combat two major issues found in standard YOLO detection:
1. **"Ghosts":** False positives generated by reflections and lens flares.
2. **"Big Arcs" & "Huge Arcs":** Massive flashes that blind the camera and are missed by YOLO because they don't look like standard arcs.

---

## 1. The Pre-Processing Masks

Before the YOLO model ever sees a frame, the image is passed through an OpenCV pre-processor. Depending on the conditions, one of three masks is applied. A mask dictates what parts of the frame YOLO is allowed to look at. Everything outside the mask is painted a solid `Cyan` color (`RGB: 0, 255, 255` | `BGR: 255, 255, 0`). The cyan color was empirically proven to reduce YOLO hallucination ("ghosting") compared to standard black masks.

### Stage 1 (Tight ROI)
This is the default state for every frame. 
- **Coordinates:** `[x: 15%, y: 18%] -> [x: 88%, y: 10%] -> [x: 76%, y: 82%] -> [x: 4%, y: 64%]`
- **Purpose:** A tight Central Region of Interest (CROI) polygon. It explicitly blocks out the bottom 25% of the frame and the far sides where arcs mathematically never occur, immediately stopping YOLO from triggering on random background artifacts.

### Stage 2 (Lenient Mask)
- **Coordinates:** Blocks only the bottom 10% of the screen.
- **Purpose:** When the system detects a potential large arc, it switches to this mask, allowing YOLO to see more of the frame.

### Stage 3 (Zero Mask)
- **Coordinates:** 100% open.
- **Purpose:** The system has confirmed a massive arc via OpenCV. The mask is dropped completely to allow YOLO to attempt a bounding box, but the system overrides YOLO anyway.

---

## 2. The Trigger Logic (The Hybrid Switch)

For every single frame, OpenCV performs an initial scan in the HSV (Hue, Saturation, Value) color space before YOLO is called.

### Detecting the "Plasma Ratio"
The system searches for a specific blue/white color spectrum representing electrical plasma:
- **Lower Bound:** `[Hue: 50, Saturation: 0, Value: 220]`
- **Upper Bound:** `[Hue: 130, Saturation: 100, Value: 255]`
The code calculates the `plasma_ratio`: how many pixels in the frame match this exact color profile compared to the total number of pixels.

### The Stage 3 (Auto-Arc) Trigger
If `plasma_ratio > 0.40` (more than 40% of the screen is covered in bright blue/white plasma), the system immediately enters **Stage 3**.
- **Action:** The mask drops to 100% open. The system hardcodes an "Auto-Arc" trigger, bypassing the need for YOLO to confirm it. This solves the "blinding flash" edge case.

### The Stage 2 (Lenient) Trigger
If the `plasma_ratio` is low, the system checks the `Value` (brightness) channel. It isolates anything brighter than `240` and finds the largest continuous bright blob (contour).
- If a blob's area is larger than the `FLASH_TRIPWIRE_AREA` (500 pixels):
  - It checks the median Hue and Saturation of the blob. 
  - If it falls within `Hue: 50-140` OR `Saturation < 80`, the system enters **Stage 2**.
- **Action:** The system drops from the Tight ROI to the Lenient Mask, giving YOLO a wider field of view to catch an abnormally large arc that might fall outside the Stage 1 boundary.

---

## 3. The YOLO Execution

Once the `stage` and its corresponding `mask` are determined, the frame is painted with the cyan background outside the mask's boundary.

```python
results = model.predict(ai_v, conf=conf_threshold, verbose=False)
```

This cyan-masked frame (`ai_v`) is what is actually fed into the YOLOv8 model. 

- **If YOLO finds an arc:** It draws a bounding box on the *original* frame.
- **If the system is in Stage 3:** A bounding box might not be drawn by YOLO (due to the blinding flash), but the system forces a positive text alert anyway: `Stage 3 (Auto-Arc: XX.X%)`.

## 4. Side-by-Side Visualization

To visually demonstrate this complex pipeline in Streamlit, the application concatenates two frames horizontally using `np.hstack`:
1. **Left Side (`report_resized`):** The original camera frame with bounding boxes and trigger status overlay.
2. **Right Side (`ai_v_resized`):** The exact cyan-masked frame that YOLO processes.

This allows the user to watch the ROI masks change dynamically from frame to frame based on the OpenCV trigger logic.
